## 一、为什么需要解密？

微信视频号的视频文件**不是标准 MP4**。微信对视频前 **128KB** 进行了 XOR 加密：

- 加密算法：**Isaac64 PRNG**（伪随机数生成器）
- 加密范围：仅文件头部前 **131,072 bytes（128KB）**
- 密钥来源：TikHub API 返回的 `decodeKey` 字段（一个整数，如 `152100969`）
- 加密后果：直接播放加密文件 → 黑屏/无法解析；传给 Kimi AI → 无法识别 → 返回截断结果（Pitfall #54）

### 加密前后对比

```
加密文件头: 00 00 00 20 66 74 79 70  ...  (ftyp 被破坏)
解密后头:   00 00 00 20 66 74 79 70  ...  (ftyp ✅ 正常 MP4 签名)
                              ^^^^
                          偏移 4-8 必须是 "ftyp"
```

## 四、解密详细流程

### Step 1: TikHub 采集（获取加密视频 + decodeKey）

```bash
tikhub_fetch.sh "https://weixin.qq.com/sph/xxxxx" --download --decrypt --output tikhub_output
```

**产出**:
- `channels_detail_{timestamp}.json` — 视频 JSON 元数据
- `channels_encrypted_{object_id}_{timestamp}.mp4` — **加密视频文件**
- JSON 中 `objectDesc.media[0].decodeKey` — 解密密钥（整数）

**decodeKey 提取路径** (`collect.py` 第 348-364 行):

```python
# 优先从顶层取
dk = str(data.get('decodeKey', ''))
# 顶层没有则从 media[0] 取
if not dk:
    media_list = data.get('objectDesc', {}).get('media', [])
    dk = str(media_list[0].get('decodeKey', '') or '')
meta['decodeKey'] = dk  # 例: "152100969"
```

### Step 2: 查找已解密文件 (`find_or_download_video()`)

```python
# collect.py 第 866-868 行
dec = find_dec()  # 查找 channels_decrypted_*.mp4
if dec and os.path.getsize(dec) > 100 * 1024:
    return dec     # 已有解密文件，直接复用
```

### Step 3: WASM 解密尝试（1st 优先级 — 当前不通）

```python
# collect.py 第 875-877 行
if wasm_decrypt(enc, decode_key, dec_out):
    return dec_out
```

调用 `/tmp/wasm_decrypt.js`（Node.js 脚本），该文件**当前不存在**。

### Step 4: Python + Playwright 解密（2nd 优先级 — ✅ 实际主力）

```python
# collect.py 第 881-887 行
dec_py = os.path.join(script_dir, 'decrypt_channels.py')
```

## 五、worker.html 详解

### 5.1 页面角色

worker.html **不是给人看的网页**，而是一个 **RPC Worker 页面**：
- 被 Playwright 以 headless 模式打开
- 在浏览器环境中加载 WASM 模块
- 通过 `window.generateKeystream()` 暴露 RPC 接口供 Node.js 调用

### 5.2 加载链

```
worker.html
  │
  ├─ <script> 配置块
  │     VTS_WASM_URL = 'wechat_files/wasm_video_decode.wasm'     // 本地 WASM 路径
  │     VTS_WASM_CDN_URL = 'https://aladin.wxqcloud.qq.com/...'  // 微信 CDN 兜底
  │
  ├─ <script src="wechat_files/wasm_video_decode_decode.js">     // 加载 WASM JS glue
  │     onerror → handleWasmScriptError()                        // 失败则切换 CDN
  │     ↓
  │     加载 wasm_video_decode.wasm                               // 编译 Isaac64 PRNG
  │     ↓
  │     Module.WxIsaac64 类可用                                   // ✅ 就绪标志
  │
  └─ <script> RPC 核心
        window.generateKeystream(decodeKey)                       // 对外接口
        → new WxIsaac64(decodeKey).generate(131072)
        → wasm_isaac_generate 回调接收密钥流
        → Base64 编码返回
```

### 5.3 关键细节

| 细节 | 说明 |
|------|------|
| **密钥流大小** | 固定 **131,072 bytes (128KB)** — 与微信加密范围一致 |
| **字节序反转** | `keystreamData.set(Array.from(wasmArray).reverse())` — WASM 输出必须 reverse |
| **超时设置** | `waitForFunction` 60 秒；`generateKeystream` 内部 30 秒轮询 |
| **CDN 兜底** | 本地 `.wasm` 加载失败时自动切换到 `aladin.wxqcloud.qq.com` |
| **堆大小** | `MAX_HEAP_SIZE = 33,554,432 (32MB)` — WASM 内存上限 |

## 六、故障排查手册

### 问题 1: Pitfall #54 — Kimi 只返回 error + 钩子类型

**现象**: `analysis_keys: ["error", "钩子类型"]`，其余 16 个字段缺失

**根因**: 解密失败 → 加密文件传给 Kimi → 无法解析

**排查步骤**:

```bash
# 1. 检查日志中的 PHASE-2 信息
grep "\[PHASE-2\]\|\[DECRYPT\]\|\[WARN\]" /tmp/pipeline_err_*.log

# 2. 确认加密文件存在且大小正常
ls -la tikhub_output/channels_encrypted_*.mp4
# 应该 > 100KB，典型值 1-70MB

# 3. 确认 decodeKey 存在
grep "decode_key" /tmp/pipeline_err_*.log

# 4. 手动测试解密
python3 decrypt_channels.py \
  --input tikhub_output/channels_encrypted_xxx.mp4 \
  --decode-key <数字> \
  --output /tmp/test.mp4

# 5. 验证 MP4 签名
file /tmp/test.mp4
# 期望: data (或 ISO Media, MPEG v4 system, 1.0)
xxd /tmp/test.mp4 | head -1
# 期望: 00000020 66747970 (ftyp 在偏移 4-8)
```

### 问题 2: Playwright `waitForFunction` 超时

**现象**: `RuntimeError: Playwright 执行失败: page.waitForFunction: Timeout...`

**可能原因及解决方案**:

| 原因 | 检查命令 | 解决方案 |
|------|----------|----------|
| Node.js 无 playwright 模块 | `node -e "require('playwright')"` | `npm install -g playwright` |
| Chromium 未安装 | `npx playwright install --dry-run` | `npx playwright install chromium` |
| 解密仓库为空壳 | `ls /tmp/WeChat-Channels-Video-File-Decryption/api-service/worker.html` | 见下方「恢复解密仓库」 |
| NODE_PATH 未设置 | `echo $NODE_PATH` | 脚本内自动设置，无需手动 |

### 问题 3: 解密仓库丢失或损坏

**现象**: `find_decryption_repo()` 返回 None 或目录为空

**恢复方法**（无需 git clone，手动下载 4 个文件即可）:

```bash
REPO="/tmp/WeChat-Channels-Video-File-Decryption"
BASE="https://raw.githubusercontent.com/Evil0ctal/WeChat-Channels-Video-File-Decryption/master"

mkdir -p "$REPO/api-service/lib" "$REPO/wechat_files"

# 下载 4 个必需文件
curl -sL "$BASE/api-service/worker.html" -o "$REPO/api-service/worker.html"
curl -sL "$BASE/api-service/lib/decrypt.js" -o "$REPO/api-service/lib/decrypt.js"
curl -sL "$BASE/wechat_files/wasm_video_decode.js" -o "$REPO/wechat_files/wasm_video_decode.js"
curl -sL "$BASE/wechat_files/wasm_video_decode.wasm" -o "$REPO/wechat_files/wasm_video_decode.wasm"

# 验证
find "$REPO" -type f \( -name "*.html" -o -name "*.js" -o -name "*.wasm" \)
# 应输出 4 个文件路径
```

> 💡 **提示**: GitHub 直连慢时可逐个文件下载（每个都是独立 HTTP 请求）。`.wasm` 文件最大（3.8MB），可能需要更长超时。
>
> 🆘 **终极兜底**: 即使 `.wasm` 文件缺失，worker.html 会自动从微信 CDN `aladin.wxqcloud.qq.com` 加载——**前提是网络能访问微信域名**。

### 问题 4: Pitfall #55 — `_clean_val` NameError

**现象**: `NameError: name '_clean_val' is not defined`

**原因**: `collect.py` 中 `_read_field_value()` 调用了 `_clean_val()` 但函数未定义

**修复**: 在 `_read_field_value()` 定义前添加函数（已在 2026-06-25 修复）:

```python
def _clean_val(val):
    """统一处理字段值：list/dict→取文本，其他保持原样"""
    if val is None:
        return ''
    if isinstance(val, list):
        first = val[0] if val else ''
        if isinstance(first, str):
            return first
        if isinstance(first, dict):
            return first.get('name', '') or first.get('text', '') or ''
        return str(first)
    if isinstance(val, dict):
        return val.get('name', '') or val.get('text', '') or json.dumps(val, ensure_ascii=False)
    return str(val)
```
